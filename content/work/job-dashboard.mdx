## Hook

I noticed the same thing every job searcher notices: the interesting roles weren&apos;t on LinkedIn, they were on company career pages. Specifically on Greenhouse and Ashby — ATS platforms that quietly expose structured job data via public APIs. The data was sitting there. The product I wanted didn&apos;t exist. So I built it.

## Why I built it

Job hunting is *retrieval and ranking*, but the tools in the market treat every listing as equally relevant. I wanted a system that filtered first by *role* (not company), scored survivors against my *actual resume*, and surfaced a small high-signal batch twice a week — not a daily firehose.

The side effect: building this taught me more about pipelines, LLM-as-judge scoring, prompt caching, and cost ceilings than reading three articles on each would have.

## How it works

Seven stages, three of them where AI matters and four of them that are just hygiene:

**Role-first triage.** The company list is a *discovery net*, kept deliberately wide (~150–300 companies). What gets through is decided by `roles.yaml`, not who&apos;s posting. That single design choice keeps the company list cheap to expand and the relevance filter sharp.

**Keyword filter before LLM scoring.** Tokens cost money. The role-filter stage drops ~95% of listings via keyword rules before anything reaches Claude. The LLM only sees the listings that *might* be a fit. This is the difference between a $5 run and a $0.50 run.

**Prompt caching for the stable inputs.** The scoring rubric and the role-specific resume are sent with `cache_control: ephemeral`. First call pays cache-write rates; the rest of the run reads from cache. The job description is the only uncached block.

**Tool-use for structured output.** The scorer doesn&apos;t free-text its answer — it calls a `emit_fit_score` tool whose JSON schema is `{fit_score: 0-100, reasons: str[], red_flags: str[], must_have_gaps: str[]}`. Pydantic validates before any row gets written.

**Cost ceiling.** Every run has a `COST_CEILING_USD` and a running cost tracker. When the next call would breach it, the loop stops cleanly and logs the remaining unscored count. The pipeline never silently spends more than I told it to.

**Google Sheets as the data store.** Cheap, durable, manually inspectable, and the Next.js dashboard reads from it without standing up a database. Underrated as a backend for personal projects.

## What I learned

**Retrieval beats generation, every time.** Eighty percent of the value came from getting the data and filtering it cheaply. The LLM-judge is the cherry; it only earns its keep on the rows that survive the keyword filter.

**Prompt caching changes the unit economics.** Sending the resume + rubric on every call without caching would have made this project too expensive to run. Caching the stable blocks dropped per-listing cost by an order of magnitude. *Token economics is a product concern,* not an FP&amp;A concern.

**Tool-use beats JSON-mode for structured outputs.** Forcing the model to call `emit_fit_score` with a strict schema eliminated an entire class of parse failures. The model can&apos;t *forget* to include `red_flags` — the tool definition won&apos;t let it.

**An eval you actually run is worth ten you wrote and forgot.** I keep a small set of listings I&apos;ve manually scored. Every prompt change runs against that set first. The 80/100 threshold on the rubric isn&apos;t magic; it&apos;s just the number where the eval set started behaving.

**The pipeline fails in boring ways, not interesting ones.** Most of the actual bugs were in scrapers — pagination, rate limits, Ashby schema drift. The "AI part" of an AI pipeline is rarely where it breaks.

## What I&apos;d build next

A "why this *isn&apos;t* a fit" mode for borderline listings — sometimes the most useful answer is a sharp rationale-backed no, not silence. I&apos;d also add a feedback loop where I can mark dashboard rows as *fit / not fit* and have the rubric tune from that signal. The infrastructure is already there; it&apos;s a UI + a write-back.
