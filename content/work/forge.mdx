## Hook

Every fitness app I&apos;ve used hands you a static plan on Monday and pretends Wednesday isn&apos;t different. But Wednesday *is* different — you slept four hours, you&apos;re sore, your heart rate variability tanked. A plan that doesn&apos;t see Tuesday is a plan that&apos;s already wrong. Forge started as &quot;what if the plan actually used the data?&quot; — and turned into a study in how much intelligence you can ship without any generation at all.

## Why I built it

Two reasons. The obvious one: I wanted a fitness app I&apos;d actually use, where the daily session reflected *today&apos;s* state, not last week&apos;s plan. The deeper one: I wanted to feel where the boundary actually sits between *AI products* and *data products with personalization*. Forge ended up being the project where I learned that the boundary is closer than people think — and a lot of what gets called &quot;AI&quot; is really good data plumbing plus deterministic rules.

Forge has no LLM. It has Apple Health, a recovery score, a generator that&apos;s a pure function of state, and a 7-day split cycle. That&apos;s it. And it&apos;s more responsive to my actual life than any app that *does* use AI.

## How it works

Three layers — read, decide, render — with localStorage as the system of record between them.

**The recovery score is a pure function.** HRV, resting heart rate, sleep, recent muscle groups → a number 0–100 plus an intensity label (low / moderate / high). No model, no API call, no inference time. It runs on-device in milliseconds. Every choice downstream — what split, how much volume, whether today is even a workout day — falls out of that score plus the cycle state.

**Active recovery is a first-class split, not a fallback.** When the recovery score comes back low (bad sleep, elevated RHR, suppressed HRV), the generator doesn&apos;t hand you a half-baked workout. It hands you a rest day — box breathing, a recovery-habits checklist (hydrate, mobility, sun), and gentle mobility work. The home screen UI swaps. The CTA changes. The whole experience reshapes around the answer instead of hiding it.

**The cycle is a 7-day state machine.** `push → pull → legs → push → pull → legs → active_recovery`. Deterministic, predictable, debuggable. The decision of *which* split to generate is independent of *how* that split gets generated — and either can be tested in isolation.

**localStorage is the system of record.** `healthData`, `userPreferences`, `appSettings`, `workoutCycle`, `workoutHistory`, `recoveryHabits` — six keys, one persistence model, all manually inspectable in Safari Web Inspector. The same logic I learned to love with Job Radar&apos;s Google Sheets backend: cheap, durable, no migrations.

**The stack is more interesting than it looks.** Next.js 16 with `output: "export"` produces a static bundle; Capacitor wraps it in a WKWebView and gives it native bridges (HealthKit, notifications, Apple Watch, in-app purchases). The whole app is one `npm run build` away from running on the web too. The trade-off: the iOS dev loop is slower than web (build → `npx cap copy ios` → Xcode → device), and React 19&apos;s error swallowing inside the WebView creates uniquely cursed debugging sessions — blank black screens with no console output. I lost a weekend to that before adopting a hard rule about always wrapping risky render paths in try/catch with hardcoded-color fallback UI.

## What I learned

**A lot of &quot;AI&quot; is good data structure plus deterministic rules.** I went into Forge thinking I&apos;d need a model to generate workouts well. I ended up not needing one — the structure of the data plus a handful of rules is more reliable, faster, cheaper, and easier to reason about. Save the model for the parts of the product that genuinely need open-ended generation. This is the project that made me skeptical of LLM-by-default.

**Personalization is mostly remembering yesterday.** The interesting product moments aren&apos;t in *today&apos;s* generation — they&apos;re in how *today* reacts to *yesterday*. Most fitness apps don&apos;t persist the right things to react to anything. Forge persists more than it generates, and that&apos;s the asymmetry that matters.

**A failure state needs to be a real state.** &quot;Active recovery&quot; isn&apos;t a graceful degradation of a workout — it&apos;s its own product surface. Treating it that way meant the UX is *better* on bad-recovery days than on good ones, because the right answer is shown loud and unambiguously. The model-failure equivalent in agentic systems: design the &quot;I don&apos;t know how to do this&quot; state as a first-class UX, not a fallback.

**Native is slow feedback by web standards.** Forty-five seconds from code change to running on device. No live reload. No reliable console output in Xcode. React 19 catches render errors internally and unmounts the tree → blank black screen with no signal. The fix is discipline you don&apos;t need on the web: ErrorBoundary at the root, try/catch around risky render paths, hardcoded-color fallback UIs that don&apos;t depend on Tailwind classes or CSS variables having loaded. The web spoils you.

**Tailwind v4 is the right cost.** CSS variables for the theme tokens, `var(--token)` inline styles for anything that must render correctly even if class generation breaks, Tailwind utilities for decoration. That split survived the React-19-eats-errors hellscape — important elements rendered even when class generation lagged.

## What I&apos;d build next

The recovery score should *learn from outcomes*. Right now it&apos;s a pure function of metrics, with no feedback loop — did the session it gave me actually fit? Did I swap half of it? Did I drop the last set? That feedback is the missing ingredient. Adding it is the moment Forge stops being a planner and starts being a coach.

After that, the place where an LLM *would* earn its keep: variation. Same target muscle group, same recovery score — give me a different feel. Different rep schemes, different equipment substitutions, different intensity techniques. The deterministic generator picks the *shape* of the workout; a small model picks the *flavor*. Generation only where it adds something rules can&apos;t.
